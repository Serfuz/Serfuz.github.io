File System

Boot Block

  • Contains machine code instructions for bootstrapping the OS.
  • Loads the OS kernel during system startup.
  • Located at the very beginning of the disk.

Superblock

  • Stores file system metadata:
    • File system type
    • Number of inodes and data blocks
    • Block size
  • Helps the OS interpret and organize files on the storage device.
  • Driver: Handles communication between OS and physical device (read/write commands).

I-bmap (Inode Bitmap)

  • Bitmap tracking free/used inodes.
  • Each bit corresponds to an index in the Inode Table:
    • 0 = free, 1 = used.
  • Limit: Number of bits = max number of files/directories.
    • Example: 64-bit I-bmap → max 64 files/directories.

D-bmap (Data Bitmap)

  • Bitmap tracking free/used data blocks.
  • Each bit corresponds to a data block index:
    • 0 = free, 1 = used.
  • Limit: Number of bits × block size = total storage capacity.
    • Example: 128 bits × 4 KB = 512 KB.

Inode Table

  • A structured collection (conceptually a “table”) of inode records.
  • Each inode describes:
    • File type (regular file, directory, etc.)
    • Permissions, owner, timestamps
    • Size
    • Pointers to data blocks (direct, indirect, multi-level indexing for large files).

Inode

  • Represents a file or directory.
  • Identified by inode number, not by name.
  • Names are resolved via directory lookups.

Data Block

  • Stores actual file content or directory entries.
  • Directory data blocks contain name → inode number mappings, not file content.
  • Block size determined by file system (e.g., 4 KB).

Key Relationships

  • I-bmap → Inode Table → Inode → Data Blocks
  • D-bmap → Data Blocks
  • Directories use data blocks for mappings, not file content.

File System Limits

  • Max files = number of inodes (I-bmap size).
  • Max storage = number of data blocks × block size.
  • You can run out of inodes before space, or vice versa.

Free Space Management

  • Bitmaps (I-bmap, D-bmap) or free lists.

File System Operations

  • Open:
    • Resolves path by reading directory inodes and data blocks recursively.
  • Read:
    • Uses inode pointers to fetch data blocks.
  • Write:
    • Updates bitmaps, inode metadata, and block pointers.

Virtual File System (VFS)

  • Abstraction layer for system calls.
  • Allows multiple file systems (ext4, FAT, etc.) under one interface.

Disk Buffer Cache

  • Speeds up reads/writes by caching frequently accessed blocks.

I/O Efficiency

  • Interrupts: Reduce CPU overhead vs polling.
  • DMA (Direct Memory Access): Transfers large chunks without CPU copying each word.

Write Scenarios

Condenced

  1. App calls write(fd, buf, size).
  2. Kernel maps fd → file object → inode.
  3. VFS dispatches to specific file system driver.
  4. File system driver updates inode and bitmaps, prepares block I/O.
  5. Generic block layer sends request to block device driver.
  6. Disk controller uses DMA to transfer data.
  7. Interrupt signals completion.

The Journey of Writing a File (AI-Copilot + update by me)

Imagine you’re saving a new file called report.txt to your disk. Here’s what happens behind the scenes:

  1. Checking the Maps
    The file system first consults the I-bmap (Inode Bitmap) to find a free inode. This bitmap is like a seating chart for inodes—each bit represents whether an inode is free or occupied. It finds an empty spot and marks it as used. Consider it as the seat is now reservered for an Inode
  2. Creating the Inode
    Next, the system creates an inode in the Inode Table at the corresponding (reserved) position found in I-bmap. This inode is the file’s identity card—it stores metadata like file type, permissions, owner, timestamps, and most importantly, pointers to data blocks where the actual content will live.
  3. Finding Space for Data
    The file system then checks the D-bmap (Data Bitmap) to locate free data blocks. These blocks are the actual storage units for your file’s content. It picks a few empty blocks and marks them as used.
  4. Writing the Content
    Your text—“Hello, world!”—is written into those data blocks. The inode’s pointers are updated to reference these blocks, so the system knows where your file’s data resides.
  5. Updating Metadata
    The inode updates its size and timestamps. The superblock might also update global counters (like free block count), ensuring the file system stays consistent.
  6. Mission Complete
    Now, when you open report.txt, the OS looks up the inode, follows its pointers to the data blocks, and retrieves your content.
